# BlogForEveryone 项目计划书

## 1. 项目背景
面向零基础用户，提供“可视化点击操作”的博客搭建与管理平台。用户无需手写命令或深入框架底层，即可完成博客初始化、主题选择、配置编辑、发布部署、跨设备迁移与 RSS 阅读订阅。

## 2. 目标与范围

### 2.1 核心目标
1. 一键式博客创建与发布（首版支持 Hexo、Hugo）。
2. 主题可视化配置：把“可配置项”全部显式展示，不让用户自行摸索。
3. GitHub 为默认发布渠道与部署承载（GitHub Pages）。
4. 本地工程可备份到独立仓库，实现换设备恢复。
5. 支持导入现有博客工程/配置并继续编辑。
6. 内置 RSS 阅读器与更新通知（应用内通知中心 + 系统通知）。
7. RSS 订阅清单可导出并与博客工程一起迁移，导入后自动恢复订阅。

### 2.2 非目标（首版不做）
1. 非静态博客平台（如 WordPress 托管）。
2. 自建云同步服务（首版仅通过 GitHub 仓库完成同步）。
3. 移动端 App。

## 3. 已确认决策
1. 产品形态：桌面应用。
2. 技术栈：Electron + Vue + JavaScript。
3. GitHub 授权：OAuth 设备码登录。
4. 首版主题数量：Hexo 5 个 + Hugo 5 个。
5. RSS 通知方式：应用内通知中心 + 系统桌面通知。
6. 首版系统支持：仅 Windows。
7. 首批主题名单：
   - Hexo: Landscape, Next, Butterfly, Fluid, Volantis
   - Hugo: PaperMod, LoveIt, Stack, Mainroad, Anatole
8. 默认发布路径：GitHub Actions 自动部署。
9. 备份仓库可见性：创建时由用户选择（公开/私有）。
10. 包管理器统一：pnpm。
11. 启动前置检测：Node.js/Git/pnpm 缺失时提供一键下载安装引导。
12. 下载失败策略：自动切换镜像源重试。
13. GitHub OAuth 设备码登录采用真实接口流程。
14. 提供 winget 静默安装模式（用户确认后自动安装 Git/Node）。
15. 提供教程中心目录化分区，覆盖关键操作流程与注意事项。
16. 图片配置默认采用“本地文件转存到工程图片目录并自动回填路径”，不强制使用图床。
17. 内容编辑需覆盖文章、关于、友链、公告四类内容，并支持创建后调用系统默认编辑器。

## 4. 需求拆解与功能设计

### 4.1 新手友好的可视化操作

1. 提供向导式流程：选择框架 -> 选择主题 -> 基础信息 -> 发布配置 -> 完成。
2. 每一步都给默认值与“推荐配置”。
3. 实时预览配置影响（标题、导航、配色、首页布局等可预览项）。
4. 启动时环境检测：若未安装 Node.js/Git，弹窗引导下载；Node 就绪后可一键安装 pnpm。
5. 环境缺失时可切换到“自动安装模式”：调用 winget 静默安装并支持失败后重试。
6. 所有关键步骤页都可跳转教程中心，并由教程中心按目录化分区统一汇总。

### 4.2 Hexo/Hugo 与主题兼容

1. 框架适配层：统一接口屏蔽 Hexo/Hugo 差异。
2. 主题适配器：每个主题维护“配置元数据 schema”。
3. 发布流水线：本地构建 + 推送到 GitHub Pages 分支/工作流。
4. 依赖与脚手架统一走 pnpm（含 pnpm dlx）。

### 4.3 主题配置项全部显式展示
1. 建立 Theme Schema Registry（主题配置索引库）。
2. 来源组合：官方文档规则 + 模板配置文件解析 + 手工补充映射。
3. UI 自动渲染表单：字符串、布尔、枚举、数组、嵌套对象。
4. 未识别项进入“高级配置”区，仍可编辑并保留。

### 4.4 底层文件管理与跨设备恢复
1. 工作区管理器：集中管理博客工程路径、框架版本、主题版本。
2. 提供“备份到另一个 GitHub 仓库”按钮：
   - 工程文件备份仓库（创建时可选公开/私有）
   - 自动提交/推送
3. 一键恢复：从备份仓库克隆并重建本地工作区。

### 4.5 外部导入与二次开发
1. 导入方式：
   - 选择现有目录
   - 导入 zip 压缩包
2. 自动识别框架类型、主题类型、核心配置文件。
3. 导入后生成兼容映射并进入可视化编辑状态。
4. 导入失败时提供诊断报告（缺文件、版本不兼容、配置语法问题）。

### 4.6 RSS 阅读与更新通知
1. 订阅源管理：新增/删除/分组/标签。
2. 拉取策略：定时轮询 + 启动时同步。
3. 阅读页：列表、全文/摘要、已读状态、收藏。
4. 通知中心：显示新增文章与来源。
5. 桌面通知：新文章到达时触发系统通知。

### 4.7 RSS 订阅可迁移文件
1. 统一文件名：subscriptions.bundle.json。
2. 存放路径：博客工程根目录下 .bfe 目录。
3. 导出与导入：
   - 导出：生成完整订阅快照
   - 导入工程时自动检测并恢复订阅
4. 支持冲突策略：覆盖/合并/仅新增。

### 4.8 内容编辑与页面维护
1. 提供“内容编辑”统一入口。
2. 新建文章自动写入 front matter 并打开默认 Markdown 编辑器。
3. 关于页、友链页、公告页使用同类流程创建与编辑。
4. 可选“保存后自动发布”，通过文件变更检测触发发布流程。

### 4.9 图片资产处理
1. 用户输入本地图片路径后，应用自动复制到工程图片目录。
2. 默认目录策略：Hexo 使用 `source/img`，Hugo 使用 `static/img`。
3. 支持指定文件名，满足重命名需求。
4. 复制完成后自动回填主题配置中的图片 URL。

## 5. 系统架构设计

### 5.1 分层
1. 桌面壳层（Electron Main）：文件系统、系统通知、命令执行、GitHub 授权。
2. 前端交互层（Vue Renderer）：向导、配置表单、阅读器、通知中心。
3. 领域服务层：
   - Framework Service（Hexo/Hugo 生命周期）
   - Theme Service（schema、配置渲染、配置落盘）
   - Publish Service（GitHub Pages 发布）
   - Backup Service（工程备份与恢复）
   - Content Service（文章/页面创建、打开编辑器、自动发布监听）
   - RSS Service（订阅、解析、通知）
4. 存储层：
   - 本地 SQLite（状态与索引）
   - 本地文件系统（博客工程与配置）
   - GitHub 仓库（发布与备份）

### 5.2 核心数据对象
1. Workspace：工程元信息、框架类型、路径、远程仓库绑定。
2. ThemeProfile：主题名、版本、schema 版本、用户配置。
3. PublishProfile：仓库、分支、Pages 策略、域名配置。
4. SubscriptionSet：RSS 源列表、同步策略、通知偏好。
5. ActivityEvent：发布记录、导入记录、拉取与通知日志。

## 6. 技术方案

### 6.1 前端
1. Vue + JavaScript。
2. 状态管理：Vue Composition API（后续可演进 Pinia）。
3. 表单与验证：基础表单校验 + 可扩展校验层。
4. 组件库：优先选择可定制方案，避免重度耦合。

### 6.2 桌面层

1. Electron IPC 定义清晰边界（命令执行、文件读写、通知、OAuth）。
2. 使用 Node 子进程运行 Hexo/Hugo 与 Git 命令。
3. 命令安全策略：白名单命令与参数校验。
4. 增加环境治理服务（Env Service）：检查 Node.js/Git/pnpm、引导下载、自动安装 pnpm。
5. 增加 GitHub Auth Service：设备码授权、令牌轮询、用户信息获取与本地会话状态管理。

### 6.3 数据与同步
1. SQLite 保存工作区索引、订阅缓存、通知状态。
2. 配置文件采用 YAML/TOML/JSON 双向转换器，保留注释与格式优先。
3. 备份仓库中保留 .bfe/metadata.json 与 subscriptions.bundle.json。

### 6.4 发布部署

1. 默认发布到 GitHub Pages。
2. 提供两种路径：
   - GitHub Actions 自动构建与部署（默认）
   - gh-pages 分支推送（可选）
3. 向导收集并校验必要信息：用户名、仓库名、分支、可选自定义域名。
4. 工作流内默认使用 pnpm 安装依赖。

### 6.5 包管理与下载容错

1. 统一包管理器：pnpm（本地开发、项目依赖安装、CI 构建一致）。
2. 若 pnpm 不存在，优先使用 corepack 激活，失败则回退到 npm 安装 pnpm。
3. 下载失败自动执行镜像源切换（npmmirror）后重试一次。
4. 关键下载流程保留执行日志，便于诊断。

## 7. 迭代里程碑

1. Electron + Vue 工程初始化。
2. 工作区创建/打开/删除。

4. 环境检查弹窗（Node.js/Git/pnpm）与安装引导。

1. Hexo/Hugo 创建向导。

### M3 发布与备份（第 6-7 周）
1. GitHub Pages 发布向导。
2. 一键发布与发布日志。
3. 备份仓库推送与恢复流程。

### M4 导入与兼容（第 8-9 周）
1. 目录/zip 导入。
2. 框架与主题自动识别。
3. 导入诊断与修复建议。

### M5 RSS 阅读与通知（第 10-11 周）
1. RSS 订阅管理与阅读界面。
2. 定时同步、变更检测、通知中心。

### M6 稳定性与发布（第 12 周）
1. 回归测试、性能优化、异常恢复。

## 9. 验收标准（对应需求 1-7）

1. 不通过命令行，用户可完成新建博客并发布到 GitHub Pages。
2. Hexo 与 Hugo 各 5 个主题可正常创建、编辑、发布。
3. 每个主题的 schema 定义项在 UI 中可见且可编辑。
4. 可将完整工程备份到独立仓库，并在新设备恢复成功。
5. 支持导入现有工程并在 UI 中继续配置与发布。
6. RSS 订阅可阅读并在源更新后收到通知。
7. 订阅快照文件与工程一起迁移，导入后自动恢复。
8. 环境缺失时可一键打开 Node.js/Git 下载，Node 可用后可一键安装 pnpm。
9. 依赖下载遇到网络问题时可自动切换镜像后重试。
10. GitHub OAuth 设备码登录流程可在应用内完成并获得登录状态。
11. 用户确认后可直接通过 winget 静默安装 Git/Node。

## 10. 风险与应对
1. 主题配置差异大、字段复杂：
   - 应对：主题适配器与 schema 版本化；高频主题优先完善。
2. 导入工程兼容性不稳定：
   - 应对：分层诊断与可回滚导入流程。
3. GitHub 发布失败场景多：
   - 应对：发布前检查与错误分级提示（权限/分支/工作流）。
4. RSS 源质量不稳定：
   - 应对：容错解析、重试策略、失效源标记。

## 11. 建议的工程结构
1. apps/desktop: Electron 主应用。
2. packages/core: 框架适配、主题 schema、发布与备份服务。
3. packages/ui: 通用 UI 组件与表单渲染器。
4. packages/rss: RSS 拉取、解析、通知策略。
5. docs/plans: 计划与设计文档。
