# Post-v1.3.5 Audit And Remediation Report

> Scope: current `main` at `v1.3.5` after Patch B release. This report ranks the remaining issues and defines a safe execution order that avoids ambiguous/destructive release operations until product, security, docs, and packaging checks are complete.

## Bottom Line

Do **not** treat the current request as one giant implementation wave. The remaining work splits into three safe waves:

1. **Wave 1 — blockers, security, and core UX**
2. **Wave 2 — docs, privacy, and release-readiness**
3. **Wave 3 — release-history cleanup only after semantics are explicit**

The request to delete prior releases and “re-release v0” is currently **too ambiguous to execute safely**.

---

## Severity-Ranked Findings

### Blocker

1. **Theme upload IPC still lacks the same managed-workspace boundary as the other theme write flows.**
   - Grounded current state: `theme:getConfig`, `theme:saveConfig`, `theme:saveLocalAsset`, and `theme:applyPreviewOverrides` already go through canonical workspace enforcement in `src/main/ipc/themeIpc.js`.
   - Remaining gap: `theme:uploadImageToGithub` still forwards raw renderer payload to `uploadImageToRepo(payload)` without the same workspace/canonical-context enforcement.
   - Why this blocks shipping: this is still a privileged path crossing the renderer/main boundary and should be hardened before any new release workflow.

### High

2. **Publish user-pages UX still reads as if the deploy repo name must be manually filled.**
   - Grounded current state in `src/renderer/src/views/PublishBackupView.vue`:
     - `resolvedDeployRepoName` already derives `${login}.github.io` for `user-pages`.
     - the UI still exposes a visible `发布仓库名称` field and readiness copy still says `待补充：填写发布仓库名称。`
   - Why this matters: it contradicts the intended beginner flow and makes the release-facing docs immediately stale.

3. **Dark-mode readability is still likely too weak for low-light use on primary workflow text.**
   - Grounded current state in `src/renderer/src/styles.css`:
     - dark shell currently uses `--shell-muted: #a3a3a3` on a very dark background.
     - broad muted-token mapping already exists, but the user is still reporting unreadable text.
   - Why this matters: this is release-quality, not post-release polish, if it affects publish/import/theme workflow copy.

### Medium

4. **README and beginner docs are stale relative to the live product.**
   - Grounded current state in `README.md`:
     - still says `当前版本：v1.1.0。`
     - no release-page hyperlink near the top.
   - Why this matters: user asked for beginner-facing docs and release entry points near the top; current docs would mislead users even if the product is fixed.

5. **Package/privacy looks structurally safer than feared, but still needs an explicit verification pass.**
   - Grounded current state:
     - `package.json` build files do not include OS `userData`.
     - `storeService.js` stores state under `app.getPath('userData')/bfe-data`.
     - `safeStorage` is used for auth ciphertext.
     - `scripts/sanitize-release.js` removes several release-time artifacts.
   - Why this matters: the structure looks good, but the user explicitly asked to confirm installers do not contain test account traces or sensitive data.

6. **Repeated repo-parser logic and large-file hotspots remain maintainability debt.**
   - Grounded current state:
     - repeated `parseGithubRepo` implementations across main + renderer.
     - large hot spots remain in `ThemeConfigView.vue`, `PublishBackupView.vue`, `publishService.js`, and `useAppShell.mjs`.
   - Why this matters: these are real follow-ups, but they are not the first things to touch in the current release sequence.

### Low

7. **Release-history cleanup is not technically blocked, but semantically ambiguous.**
   - Current open ambiguity:
     - which releases to delete
     - whether tags should also be deleted
     - what “re-release v0” means exactly
     - what rollback/history should remain visible
   - Why this stays low for now: the correct action is to defer, not improvise.

---

## External Guidance Applied

- **WCAG 2.1 / accessibility guidance**: contrast must be checked under the actual dark appearance, not just inferred from palette intent.
- **Electron security guidance**: privileged IPC boundaries and exposed APIs should be minimized and sender/context should be validated for privileged flows.
- **Electron packaging/privacy**: use `safeStorage`, keep user data outside packaged files, and inspect final artifacts rather than assuming.
- **GitHub release management**: deleting a release is separate from deleting the tag; destructive cleanup should only happen with explicit semantics.

---

## Recommended Execution Order

### Wave 1 — Blockers, Security, Core UX

Ship these before docs/release cleanup:

1. **Theme upload-path hardening**
   - Bring `theme:uploadImageToGithub` under the same managed-workspace/canonical-context enforcement as the other theme write handlers.

2. **Publish user-pages UX simplification**
   - Stop visually asking the user to fill a deploy repo name for the `user-pages` path.
   - Treat `${login}.github.io` as the direct visible target.

3. **Dark-theme readability recalibration**
   - Re-tune low-light text colors for primary workflow text using a darker-shell-specific contrast pass grounded in WCAG-style thresholds and runtime verification.

### Wave 2 — Docs, Privacy, Release Readiness

Only after Wave 1 is stable:

4. **README + beginner docs refresh**
   - update versioning and flow descriptions to match the actual app
   - add a release-page hyperlink near the top

5. **Package/privacy verification**
   - inspect built artifacts for sensitive data, test-account remnants, fixture data, and local state leakage

### Wave 3 — Release Operations / Cleanup

Only when Waves 1–2 are fully green **and** release semantics are explicit:

6. **Release/tag cleanup (if still wanted)**
   - require an explicit deletion matrix before doing anything destructive
   - if semantics remain ambiguous, defer and ship a normal forward release instead

## Immediate Next Step

Start **Wave 1A** as the highest-priority safe fix:

- Harden `theme:uploadImageToGithub` in `src/main/ipc/themeIpc.js`
- Add/extend focused IPC tests proving the path now rejects unmanaged context and rewrites to canonical managed workspace data
- Keep the patch scoped to the boundary fix only

Then proceed to Wave 1B/C once that blocker is green.

## Non-Goals For The Immediate Wave

- Do not redesign the whole shell.
- Do not start broad refactors of giant files before the blocker fixes are complete.
- Do not widen backend publish/import contracts unless evidence forces it.
